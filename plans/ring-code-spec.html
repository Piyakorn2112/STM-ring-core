STM Ring-Code — Implementation Spec

A scannable, on-brand visual code: the STM ring drawn as curved dashes that bloom into the living ring when scanned.

Status: design / pre-build Whole ring in view Rotation-safe Not mirror-safe (by choice) Close-up · screen → camera Payload = id pointer
One-paragraph thesis. The ring is a one-way visualization of an id, so the geometry can't be "decoded" — instead we add an explicit, decodable channel by breaking the stroke into curved dashes along the main arc (the look in your sketch: an incomplete flower / dust of the ring). Those dashes carry a tiny payload — a serial / command id — which the backend resolves. On a good scan we render the real living ring for that id and morph the dashes into it, registered onto what the camera sees. That bloom is both the verification and the brand moment.

1Concept & visual language

The code-ring is the same centre-line as the brand ring (so it reads as "the STM ring"), but its stroke is rendered as discrete dashes that curve along the arc — like the sub-strands fragmented into beads of light. At rest it is the "dust of the ring"; on a successful scan it reconstructs into the full, continuous, glowing wire.

Reference points

  • WeChat code / App Clip — a recognisable branded ring that is the scannable.
  • Difference: ours bends the code along the ring's own strands, and the "reveal" is a full generative animation, not a logo pop.

Two visual tiers (one knob)

  • Overt — crisp dashes, clear quiet zones, easy for a plain CV reader. Default for functional scans.
  • Subtle — denser, more "dust", needs the learned reader. For decorative surfaces.

2System architecture

One shared geometry core feeds two renderers (continuous brand ring + discrete code ring). A codec turns ids into symbol streams; a decoder turns camera frames back into ids; a verifier cross-checks the shape; the backend resolves ids to commands.

ModuleResponsibility
ringGeometry.ts (new, factored)The shared, brand-rule-bound centre-line: epicycle sum → arc-length resample → fair points, plus a deterministic self-crossing finder. Pure, framework-free.
stmRingCore.ts (unchanged)Existing continuous renderer + exporters. Re-points its geometry at ringGeometry.ts via re-export only (no behaviour change).
ringCodeCore.ts (new)Discrete renderer: lays dash glyphs along the arc, honours quiet zones / crossing keep-outs, draws the fiducial. Emits the code-ring SVG + a "bloom" target frame.
ringCodec.ts (new)Pure data: encode(id) → symbols and decode(symbols) → id. Framing, ECC, symbol mapping. No DOM, no pixels.
ringDecoder.ts (new)Vision: camera frame → ring detection → trace/unwrap → symbol sampling → hands symbols to the codec.
ringVerify.ts (new)Cheap invariant fingerprint of the captured shape (Fourier-descriptor + crossing signature) vs the decoded id's expected fingerprint.
Scanner UI (new route)Camera preview, alignment guidance, the dash → ring bloom, pass/fail theatre.

3Code-ring visual spec

3.1 Geometry — the "code profile"

The code-ring draws from a restricted, calmer subset of the existing shape family, all still inside the brand rules, so there is plenty of clean arc to write on:

  • Symmetry OFF — rotational symmetry creates a unique-start ambiguity; fatal for a scanner.
  • Low k1 (fewer loops) and no third epicycle (a3 = 0) — fewer self-crossings, longer continuous runs.
  • Avoid the knot / high-lobe archetypes; prefer classic / light rosette.
  • The shape seed is hash(payload-id) through this restricted generator, so the silhouette still authenticates the id (one-way), while always being code-friendly.

3.2 The substrate — arc, lanes, slots

Track
The main loop's arc-length, 0 → L (perimeter), in winding order.
Slots
N equal arc-length cells around the loop (target N = 32).
Lanes
Optionally 2 radial sub-strands (inner / outer) per slot — the parallel curved sub-lines in your sketch — for +1 bit.
Glyph
Per slot: a curved dash whose presence + length (and lane / colour) encode the symbol.
Quiet zones
A clear gap each side of the fiducial and around crossings, so boundaries are unambiguous.

3.3 The fiducial (absolute origin + direction)

Reuse the charge arc. The brand ring already has a vivid, position-controlled "charged" region. Make it the fiducial: a unique orange solid comet sector (thick → thin) placed at slot 0. It is colour-unique (data dashes never use orange), shape-unique (solid, not dashed), and asymmetric so it fixes both the origin and the winding direction in one feature. Because we don't need mirror-safety, that's all the orientation logic required.

3.4 Crossing keep-outs (the overlap problem)

Self-crossings occlude data and confuse tracing. We don't fight them:

  1. Compute every crossing analytically from the ordered points (segment–segment intersection) — exact, deterministic, in arc-length coordinates, with over/under known from draw order.
  2. Reserve a keep-out interval ±margin around each crossing; the slot layout skips those (a 1-D interval-packing on the cycle).
  3. Exploit them: the over/under weave keeps the top strand continuous through a crossing, so the tracer follows it; crossings double as registration landmarks.

4Data & frame format

Whole-ring-in-one-frame means no segment indexing: one fiducial, one payload run, one whole-loop ECC, read in a single pass.

// Logical frame, read clockwise from the fiducial
[ FIDUCIAL ] [ VER 2b ] [ PAYLOAD 24–37b ] [ ECC Reed–Solomon ] [ quiet ]
        slot0      ver       the id pointer            parity         wrap

// Payload is an opaque id; the backend maps id → command.
// Optional: PAYLOAD = id ++ HMAC(id) for signed/anti-spoof codes.

Verdict: ~25–37 payload bits is plenty for a serial / command id, especially since the backend holds the real meaning and can remap it live. Aim for N = 32, 2 bits/slot in v1; bump to 3 bits or lanes if you need headroom.

Two implementations, same interface. Start with classical CV (segment → skeleton → trace → sample). Free labels are this project's superpower.

7Verification & the scan experience

7.1 Layer A — cheap shape check (machine)

After decoding id, confirm the captured shape really is that id's canonical render — without expensive pixel matching:

  • Fourier-descriptor fingerprint — the curve is an epicycle sum, so a handful of low-order boundary harmonic magnitudes are a rotation/scale-invariant signature, almost free to compute and highly discriminative.
  • Crossing signature — count + sorted arc-gaps between self-crossings; also invariant, already computed during tracing.
  • Compare measured vs the id's expected fingerprint within tolerance → reject spoofs / mis-reads before trusting the id.

7.2 Layer B — the bloom (the actual product)

On success, render the live ring for the decoded id and morph the captured dashes into it, registered onto the scanned ring using the fiducial + crossing landmarks. Reuse the existing engine: buildRingMorph / the settled-pose machinery / the charge arc igniting and drifting. Dust → flower → fully-formed glowing ring.
Scanner stateVisual
SearchingLive camera; faint predicted ring ghost that snaps/locks when alignment is good (on-brand replacement for QR corner-finding).
Locked + verifiedDashes bloom into the continuous living ring; charge arc lights; subtle settle. = "authentic".
Decoded but failed Layer ARing tries to form, then glitches / refuses to settle. = "not genuine / try again". Human-legible verdict, no text needed.

Because payload is just an id, decode is fast enough that the bloom feels instant — the experience carries the perceived reliability.

8Backend & realtime sync

  • The code is a stable pointer; semantics live server-side. id → command can be remapped without reprinting/redisplaying.
  • Both displayer and reader are your web apps on the same (or linked) backend → resolution + state sync over the existing channel; a scanned id can trigger a live action and reflect back on the display in real time.
  • Optional signed codes: payload = id ++ HMAC(id) so only your backend-issued ids verify — pairs with the Layer-A shape check for tamper-evidence a plain QR can't offer.
